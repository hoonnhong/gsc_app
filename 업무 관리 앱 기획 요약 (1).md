# **업무 관리 앱 개발 방향성 및 기획 요약**

## **1\. 프로젝트 개요**

* **목표:** 회계, 직원, 문서, 강사 관리 등 다양한 비즈니스 기능을 통합한 맞춤형 업무 관리 앱 개발  
* **사용자 계획:** \* **1단계:** 대표자(본인) 1인 단독 사용 (핵심 업무 자동화 및 뼈대 구축)  
  * **2단계:** 전 직원 사용 (권한 분리 및 협업 환경으로 확장)  
* **기술 스택:** Python, NiceGUI, Pandas, Google Sheets(다중 연동), BigQuery, AI(보조 도구)

## **2\. 앱 아키텍처 및 구조 전략**

* **단일 모듈형 앱 (Single Portal, Modular Design):**  
  * 여러 개의 앱을 쪼개는 대신, \*\*하나의 큰 메인 프레임(포털)\*\*을 만듭니다.  
  * 좌측/상단에 고정된 메뉴바를 두고, 기능별 코드(회계, 인사 등)는 별도 파일로 분리하여 메인 앱에 결합(Import)하는 방식을 취합니다. 관리가 훨씬 용이해집니다.  
* **메뉴 UI/UX 레이아웃 전략 (3\~4단계 복잡한 메뉴 대응):**  
  1단계를 좌측 사이드바로 고정할 경우, 하위 메뉴의 깊이(Depth)를 감당하기 위해 다음 중 하나의 UI 패턴을 앱의 표준으로 채택합니다.  
  * **타입 A (전체 아코디언형):** 1\~4단계를 모두 좌측 사이드바에 넣고 클릭 시 아래로 트리(Tree)처럼 펼쳐지는 방식. (구조 파악 용이 / 사이드바 길어짐 주의)  
  * **타입 B (L자 분리형 \- 추천):** 1단계(대메뉴)는 좌측 사이드바, 2단계(중메뉴)는 메인 화면 상단의 탭(Tab), 3\~4단계는 화면 내 서브 메뉴로 배치. (화면이 가장 깔끔하고 인지하기 쉬움)  
  * **타입 C (탐색기 카드형):** 1단계만 좌측에 두고, 2\~4단계는 중앙 화면에서 큼직한 폴더/카드 버튼을 눌러 타고 들어가는 방식. 상단에 이동 경로(Breadcrumb) 표시. (가장 직관적이고 초보자 친화적)  
* **동적 메뉴 시스템 (Dynamic Menu):**  
  * 상위 메뉴 \- 하위 메뉴 \- 더 하위 메뉴 등 **3\~4단계 이상의 깊이(Depth)를 가진 복잡한 메뉴 구조**도 유연하게 관리하기 위한 전략입니다.  
  * 앱 내부 코드로 메뉴를 고정하지 않고, **구글 시트에 메뉴 목록과 권한을 작성**해 둡니다. parent\_id를 꼬리에 꼬리를 무는 방식으로 연결하면 몇 단계의 하위 메뉴든 무한히 확장할 수 있습니다.  
    * **\[메뉴 설정용 구글 시트 컬럼 및 속성 정의\]**

| 컬럼명 (필드명) | 데이터 속성 | 설명 및 작성 예시 |
| :---- | :---- | :---- |
| menu\_id | 텍스트 (고유값) | 각 메뉴를 식별하는 고유 ID (예: M01, M01-01, M01-01-01) |
| parent\_id | 텍스트 | 직속 상위 메뉴의 menu\_id 입력. (예: 3단계 메뉴면 2단계 ID 입력, 최상위면 빈칸) |
| menu\_level | 숫자 | 메뉴의 깊이 단계 (예: 대메뉴 1, 중메뉴 2, 소메뉴 3 \- 시트 가독성 용도) |
| menu\_name | 텍스트 | 화면 메뉴바에 실제로 표시될 이름 (예: 회계 관리, 급여 대장) |
| icon | 텍스트 | NiceGUI에서 지원하는 Material Icon 이름 (예: account\_balance, people) |
| route | 텍스트 (URL 경로) | 메뉴 클릭 시 이동할 앱 내부 경로 (예: /accounting, /hr/salary) |
| allowed\_roles | 텍스트 (콤마 구분) | 해당 메뉴에 접근 가능한 권한 그룹 (예: admin, admin,staff, all) |
| sort\_order | 숫자 | 메뉴가 위에서부터 정렬될 순서 (예: 10, 20 \- 나중을 위해 10단위 추천) |
| is\_active | Boolean (TRUE/FALSE) | FALSE일 경우 데이터를 지우지 않고도 앱 화면에서 임시로 메뉴를 숨김 처리 |

  * 앱 실행 시 구글 시트를 읽어와 좌측 사이드바 메뉴를 자동으로 생성하게 만들어, 추후 메뉴 변경 시 코드 수정 없이 시트만 수정하면 되도록 유연하게 구축합니다.  
* **UI와 로직의 분리:**  
  * 화면을 그리는 코드(NiceGUI)와 데이터를 계산하는 코드(Pandas)를 분리하여, 추후 직원용 화면을 새로 만들 때 계산 로직을 재사용할 수 있게 설계합니다.

## **3\. 데이터베이스 활용 전략 (다중 소스 관리)**

* **Google Sheets (다중 파일 연동 및 분리):**  
  * 업무 성격과 보안에 따라 여러 개의 구글 시트 파일로 분리하여 운영합니다. (예: 메뉴설정\_시트, 회계\_시트, 직원관리\_시트)  
  * **앱 연동 전략:**  
    1. **환경변수(.env) 활용:** 파이썬 코드 내에 각 시트의 고유 ID를 분리해서 저장합니다. (예: ACCOUNTING\_SHEET\_ID \= '1a2b3c...')  
    2. **단일 서비스 계정:** 파이썬 앱이 구글 시트에 접근할 때 사용하는 '서비스 계정 이메일' 하나를 생성한 뒤, 우리가 사용하는 모든 구글 시트에 이 이메일을 '편집자'로 공유해 둡니다. 앱은 알아서 필요한 시트만 골라 데이터를 가져옵니다.  
* **BigQuery:** \* 구글 시트로 감당하기 어려운 대용량 데이터나, 절대 변조/삭제되면 안 되는 핵심 데이터(예: 수년 치 회계 원장, 대규모 로그 데이터)를 적재합니다. 초기부터 BQ와 시트를 혼용하여 구조를 잡습니다.  
* **Pandas:** \* 여러 구글 시트와 BigQuery에서 흩어진 데이터를 불러와, 앱 내부에서 병합(Merge)하고 가공, 통계, 필터링하여 NiceGUI 화면에 출력합니다.

## **4\. 각종 보고서(인쇄 및 저장) 구현 전략**

웹 환경에서 가장 까다로운 인쇄 및 문서 포맷팅을 최소한의 코딩으로 해결하기 위한 4가지 맞춤형 전략을 사용합니다.

* **Track 1: 단건 서식 출력 (구글 시트 템플릿 주입 방식)**  
  * **용도:** 지출결의서, 급여명세서, 증명서 등 특정 A4 양식이 필요한 문서.  
  * **방법:** 구글 시트에 인쇄용 서식을 만들고, 파이썬으로 특정 셀에 값만 업데이트한 뒤 PDF로 다운로드하게 합니다.  
* **Track 2: 대량 데이터 출력 (Pandas to Excel 방식)**  
  * **용도:** 월별 회계 장부, 전체 강사 목록 등 표(Table) 형태의 데이터.  
  * **방법:** 화면에 띄워진 Pandas 데이터를 to\_excel() 함수를 통해 즉시 엑셀 파일로 변환하여 사용자 PC로 다운로드 시킵니다.  
* **Track 3: 화면 즉시 인쇄 (NiceGUI \+ CSS Print)**  
  * **용도:** 현재 보고 있는 통계 차트나 대시보드 화면.  
  * **방법:** window.print()를 활용하되, CSS @media print로 사이드바와 버튼을 숨기고 본문만 출력합니다.  
* **\[추천\] Track 4: AI 기반 HTML 템플릿 렌더링 (Excel to HTML 방식)**  
  * **용도:** 앱 내부에 완전히 내장하여 빠르고 예쁘게 보여줘야 하는 고정 서식(위촉장, 수료증 등).  
  * **방법:** 1\. 엑셀에서 원하는 양식을 예쁘게 디자인합니다. 2\. 엑셀을 '웹 페이지(HTML)' 형식으로 저장합니다. 3\. 저장된 HTML 코드를 AI(Gemini, ChatGPT 등)에게 주고, "이 HTML을 NiceGUI에서 사용할 수 있도록 이름, 금액 등 변수가 들어갈 자리를 {{ name }}, {{ amount }} 형태로 바꿔서 정리해줘"라고 요청합니다. 4\. 정리된 HTML 템플릿 코드를 NiceGUI의 ui.html()에 띄우고 브라우저 인쇄 기능을 활용합니다. 5\. HTML/CSS를 직접 짜는 고통 없이 앱 안에 완벽한 서식을 내장할 수 있는 최고의 효율 방법입니다.

## **5\. 개발 및 유지보수 관리 (Issue Tracker)**

전문 개발 회사의 시스템을 차용하여, 앱 개발 과정에서 발생하는 모든 요구사항, 에러, 아이디어를 **별도의 구글 시트(개발\_관리\_시트)로 통합 관리**합니다.

* **운영 방식:** \* 머릿속에 있는 아이디어나 발견한 버그를 즉시 시트에 기록합니다.  
  * 우선순위가 높은 것부터 하나씩 개발/수정하며 상태를 변경합니다.  
  * 추후 직원들이 앱을 사용할 때 "피드백/버그 신고" 버튼을 만들어, 이 시트에 자동으로 접수되도록 연결할 수 있습니다.  
* **\[개발 관리 시트 컬럼 정의\]**

| 컬럼명 | 데이터 속성 | 설명 및 작성 예시 |
| :---- | :---- | :---- |
| issue\_id | 텍스트 (자동) | 고유 번호 (예: REQ-001, BUG-005) |
| 유형 | 드롭다운 | 기능추가, 에러수정, 디자인수정, 단순아이디어 |
| 상태 | 드롭다운 | 대기, 진행중, 완료, 보류 |
| 우선순위 | 드롭다운 | 긴급, 높음, 보통, 낮음 |
| 제목 | 텍스트 | 이슈의 핵심 요약 (예: "로그인 버튼 클릭 시 무반응") |
| 상세내용 | 텍스트 | 어떤 화면에서 어떤 문제가 있는지, 또는 어떤 기능이 필요한지 상세히 기재 |
| 요청자 | 텍스트 | 본인 또는 피드백을 남긴 직원 이름 |
| 등록일 / 완료일 | 날짜 | 이슈가 등록된 날짜와 해결된 날짜 |

## **6\. 단계별 개발 로드맵 (애자일 방식)**

### **\[Phase 1\] 1인 사용 및 핵심 기능 집중**

* 복잡한 로그인/권한은 뒤로 미루고, 본인의 실무 시간을 줄여줄 가장 핵심적인 기능(예: 급여 계산, 회계 장부) 1\~2개부터 개발합니다.  
* UI 디자인에 얽매이지 않고, 데이터를 엑셀처럼 띄워놓고 빠르게 처리하는 데 집중합니다.  
* 전체 앱의 뼈대(메인 화면 \+ 동적 사이드바)를 우선 구축합니다.

### **\[Phase 2\] 직원 배포 및 권한 관리 (RBAC) 도입**

* 기능이 안정화되면 로그인 시스템을 도입합니다.  
* 구글 시트의 메뉴 설정에 '접근 권한' 열을 추가하여, 관리자(본인)는 모든 메뉴를 보고 일반 직원은 본인에게 허락된 메뉴(근태, 본인 문서 등)만 보이도록 제어합니다.  
* 일반 직원이 사용할 화면은 입력 실수나 교육 시간을 줄일 수 있도록 직관적이고 단순한 UI로 별도 구성합니다.
